## 2. 軟體開發生命週期（SDLC）發生劇變

- AI 代理正在從根本上改變軟體開發的每一個環節
- 撰寫、除錯、維護程式碼等**戰術性工作**加速轉移至 AI
- 工程師的核心價值聚焦於**架構設計、系統設計與策略決策**
- 這不僅是工具升級，而是整個職業角色的重新定義

### Traditional SDLC vs. Agentic SDLC

| 面向 | Traditional SDLC | Agentic SDLC |
|------|-----------------|--------------|
| 週期時間 | Weeks-Months per cycle | Hours-Days per cycle |
| 需求與規劃 | Days-Weeks | Express Intent（Minutes） |
| 系統設計 | Weeks | Agent Understands（Seconds） |
| 實作與編碼 | Weeks-Months | Agent Implements（Minutes） |
| 測試與 QA | Days-Weeks | Agent Tests + Docs（Minutes） |
| Code Review | Days | Human Review（Min-Hours） |
| 部署與發布 | Days | Deploy and Ship（Minutes） |
| 監控與可觀測性 | Ongoing | Ongoing + auto fix |
| 回饋與迭代 | Continuous | Learn and Iterate（Ongoing） |

### Key Differences

| 傳統 | Agentic |
|------|---------|
| Sequential handoffs | Fluid agent flow |
| Human codes everything | Human guides, agent executes |
| Docs as afterthought | Docs generated inline |
| Manual incident response | Agent-assisted remediation |

## 3. 單一代理進化為協調式團隊

- 如同軟體團隊從個人英雄式開發走向敏捷協作，AI 代理也正從單兵作戰進化為**多代理協調式工作流程**
- 需要全新技能組合：**任務分解、代理專業化、協調協議設計**

### 三大關鍵能力

1. **任務分解** — 將複雜需求拆解為可平行執行的子任務，分配給專業化的代理
2. **階層式調度** — 主控代理協調多個子代理，監控進度並整合結果
3. **同步與版控** — 開發環境需處理多個並行代理的狀態與同步貢獻

### Single Agent vs. Multi-Agent Architecture

| Single Agent | Multi-Agent Teams |
|-------------|-------------------|
| Sequential bottlenecks | Parallel processing |
| Single perspective, blind spots | Diverse views catch issues |
| Context window limits scope | Distributed context capacity |
| General-purpose reasoning | Role-specific specialties |
| Minutes-to-hours tasks | Days-to-weeks projects |

### 💡 實戰案例：Fountain

- 透過 Claude 的多代理階層式調度：
  - 篩選速度提升 50%
  - 入職速度加快 40%
  - 候選人轉換率提升 2 倍
  - 一家物流客戶將新倉庫人員配置時間從一週以上縮短至 72 小時內

## 5. Skill vs. Sub-Agent

> Agentic Engineering 架構中的兩種能力擴充模式

### 核心問題

> 「這個操作需要自己決定『下一步做什麼』嗎？」

### 比較表

| 面向 | Skill | Sub-Agent |
|------|-------|-----------|
| 本質 | 確定性工具／函式 | 自主推理的獨立 Agent |
| 控制流 | Caller 完全掌控 | 有自己的 agentic loop |
| 決策 | 無，純執行 | ✓ 自行 plan → act → observe |
| 狀態 | Stateless | 可持有 context / memory |
| 接口 | 明確 schema（參數） | 高層次 goal（自然語言） |
| 可驗證性 | 高（deterministic） | 低（probabilistic） |

### 架構示意

```
Orchestrator
 ├── calls → Skill A (tool: fixed logic)
 ├── calls → Skill B (tool: fixed logic)
 └── delegates → Sub-Agent (goal-driven)
      ├── calls → Skill C
      └── calls → Skill D
```

### 設計原則

- **能用 Skill 就不用 Sub-Agent** — 避免引入不必要的 Autonomy，簡化系統複雜度
- **Skill → 納入 Eval Harness** — 讓工具功能可被高度模擬、測試與重現
- **Sub-Agent → 對應凍結的 Sub-task Spec** — 每個子代理應有明確的任務定義，如同 SDD 的委派單元
- **Autonomy ↑ → Unpredictability ↑ → Harness 成本 ↑**

## 6. DDD (Domain-Driven Design) Example for Agentic Abstraction

### DDD 四層架構

| 層級 | 內容 | 說明 |
|------|------|------|
| **Domain 層（核心）** | Entities / Value Objects / Aggregates | 禁止外部依賴 |
| **Application 層** | DTOs / Use Cases / Validators | 協調領域物件 |
| **Infrastructure 層** | EF Core / Repository 實作 / 外部 API | 資料庫與第三方服務串接 |
| **API 層** | Controllers / Middleware | 不含業務邏輯 |

- **依賴方向**：API → Infrastructure → Application → Domain
- **關鍵規範**：PascalCase 命名、async/await 強制、Constructor Injection 優先、Entity 屬性 private set

### AGENTS.md 範例結構

- `## Project Overview` — DDD 分層架構的電子商務後端系統
- `## Tech Stack` — .NET 8 SDK, C# 12, SQL Server 17, EF Core, AutoMapper, FluentValidation, NUnit, Moq
- `## Architecture & Directory Structure` — `/src/Domain`, `/src/Application`, `/src/Infrastructure`, `/src/API`
- `## Coding Conventions` — Naming, Async, DI, Entity Guidelines
- `## Testing Protocols` — Unit Tests, Mocking, Commit Rule
- `## Pull Request Guidelines` — Conventional Commits, 依賴原則檢查

## 7. Subagent vs Agent Teams in Claude

### Subagents（舊模式）

- Main Agent 派出 Sub A / Sub B / Sub C
- 每個 Subagent 只能**回報給 Main Agent**（單向）
- Sub 之間**無法互通**

### Agent Teams（新模式）

- **Team Lead** — 協調、指派、整合
- **Shared Task List** — 依賴追蹤、自動解鎖、File lock 防碰撞
- **Teammate A/B/C** — 後端開發 / 前端開發 / 測試 QA
- **Mailbox 點對點互傳訊息**
- 需要 Claude Code v2.1.32+，Opus 4.6 以上模型

> **關鍵差異：Teammate 可以互相溝通**，不須透過 Team Lead 中轉

## 8. 從需求討論到 Agentic Coding

> Claude Teams · Claude Code · agency-agents — 從自然語言對話出發，走到 Multi-Agent 自動開發團隊的完整實作課程

### 三大核心觀念

- 💬 **對話即設計** — 不需要先寫規格，讓對話成為需求探索的過程
- 🔒 **安全是對話議題** — 與 Claude 角色切換，比事後 code review 更早發現問題
- 🤖 **Agent 是分工不是魔法** — 明確 handoff 設計與 Quality Gate 才是多 agent 的核心價值

### 8.1 人機對話建立需求

- 從一句不完整的需求描述開始，Claude 主動追問釐清
- 對話中逐步確認：功能邊界、權限分層、回傳格式、錯誤處理
- 💬 起始提示範例：「我想開發一個在 xxx API，可用來驗證文件憑證。」
- 🎯 本階段輸出：4 支端點初步共識
  1. `POST /sign` — 為 PDF 加入憑證
  2. `POST /verify` — 驗證 PDF 有效性
  3. `GET /status/{id}` — 查詢簽署任務狀態
  4. `POST /batch` — 批次簽署多份文件

> 不需要先寫規格 — 讓對話本身成為需求探索的媒介，文件是對話的沉澱，不是起點。

### 8.2 請 Claude 生成規格文件

- 對話達成初步共識後，指示 Claude 將討論結果轉化為結構化的 Markdown 規格文件
- 搭配 Mermaid 流程圖讓整體邏輯視覺化
- **文件是討論的基礎，不是最終答案** — 每一版都可以說「加入 X」或「調整 Y」來持續疊加演進

#### 版本演進

1. **v0.1.0** — 基礎 4 支端點定義，含請求/回應格式與 HTTP 狀態碼說明
2. **v0.2.0** — 加入 MCP Server 規劃，定義 tool 介面與路徑白名單
3. **v0.3.0** — 加入 Batch API 設計，含非同步任務佇列與 Redis Job Store
4. **v0.4.0** — 整合 Mermaid 流程圖、OpenAPI schema 草稿與版本變更日誌

> 關鍵提示技巧：每次修改只說「加入 MCP Server 規劃」或「調整 batch 的錯誤回傳格式」，讓 Claude 在既有版本上疊加，而非重寫整份文件。

### 8.3 Security by Design 檢核

- 規格文件完成後，**立即切換 Claude 的角色為資訊安全專家**
- 在設計層面就找出潛在漏洞，比事後 code review 更早、更全面
- 🎭 角色切換提示：「請以資訊安全專家角色，對這份需求文件進行完整的威脅建模與安全檢核」
- Claude 主動識別出 20 項安全議題，最終擴充為 **33 項檢核清單**，產出 `security_check.md`

#### 安全議題分級

| 等級 | 範例 |
|------|------|
| 🔴 高風險 | Hash 碰撞攻擊防護、API 金鑰洩漏與輪換機制、任意檔案上傳漏洞 |
| 🟡 中風險 | Rate Limiting 設計、Batch 任務權限隔離、Redis 資料加密策略 |
| 🟢 低風險 | 日誌敏感資訊遮罩、回應標頭安全設定、錯誤訊息資訊洩漏 |

### 8.4 設計 Multi-Agent 開發團隊

- 借助開源專案 `github.com/msitarzewski/agency-agents` 提供的 144 個預定義角色
- 請 Claude 根據本專案需求挑選最適合的組合

| Agent 角色 | 來源路徑 | 職責說明 |
|-----------|--------|---------|
| 🗂 Agents Orchestrator | specialized/ | 任務分派、sprint 規劃、協調各 agent 間的工作流程與優先順序 |
| 🏗 Backend Architect | engineering/ | FastAPI 架構設計、Batch API 規劃、OpenAPI schema 生成 |
| 🔐 Security Engineer | engineering/ | SEC-* 安全需求實作、威脅建模、security_check.md 對應修補 |
| 🗄 DB Optimizer | engineering/ | 資料庫 Schema 設計、Redis Job Store 配置與效能調優 |
| 🔧 MCP Builder | specialized/ | MCP tool 定義、路徑白名單管理、tool 介面規格撰寫 |
| 🧪 API Tester | testing/ | 依據 security_check.md 自動執行安全測試與回歸測試 |
| ✅ Reality Checker | testing/ | 每個 sprint 結束的品質關卡，確認實作符合規格與安全基準 |

> 明確的分工設計，比開越多 agent 更重要。

### 8.6 核心概念總結

| 概念 | 說明 |

| 💬 **對話驅動設計** | 先對話，再沉澱文件。文件是對話的副產品，不是起點。用 30 分鐘對話，讓 Claude 的追問幫你找出遺漏。 |

## 10. 可驗證性：AI 時代真正的分界線

> 整理自 Andrej Karpathy《No Priors》訪談：*The End of Coding*

### 10.1 核心命題

> AI 能自主優化模型，卻說不出一個新笑話 — 這條分界線，才是人類的護城河。

| 現象 | 根本原因 | 結論 |
|------|---------|------|
| AI 一夜超越人類兩個月的調參成果 | 有客觀指標（loss）可以驅動優化迴圈 | 可驗證 → AI 複利 |
| AI 四年來一直說同一個笑話 | 「好不好笑」無法被量化，沒有優化目標 | 不可驗證 → AI 凍結 |

### 10.2 Karpathy 案例拆解

#### ① Dobby 家庭 AI

- 用三個提示讓 Agent 掃描局域網、反向工程 Sonos API、播放音樂
- 最終整合燈光、HVAC、百葉窗、泳池、攝影機共六個 App 的功能
- **關鍵條件**：「播放有沒有成功」一試即知 → **可驗證**

#### ② AutoResearch 超參數搜尋

- 人工調參兩個月的成果，AI 跑一晚即超越
- 找到 weight decay 設定、Adam betas 組合等人類未曾察覺的改善點
- **關鍵條件**：loss 是明確的成功指標 → **可量化**

#### ③ microGPT（200 行 Python）

- AI 能完整讀懂並理解每個設計決策
- 但 AI **無法自己創造出這個問題框架本身** — 從無到有的抽象提煉，不在它能力範圍之內
- **關鍵條件**：「提煉出正確抽象」沒有事先存在的標準 → **不可驗證**

### 10.3 核心分析框架：可驗證性判斷

一個任務能被 AI 自主優化的前提，必須同時滿足三個條件：

1. **可觀察 Observable** — 能看到執行結果，輸出是明確可感知的
2. **可量化 Measurable** — 有數值或明確標準可以衡量好壞優劣
3. **可驗證 Verifiable** — 能判斷「做對了」還是「看起來做對了」

| 任務 | 可驗證？ | 建議 |
|------|---------|------|
| 資料清理與格式轉換 | ✅ 是 | 全部交給 Agent，設計自動迴圈 |
| 程式碼效能優化 | ✅ 是 | 讓 AI 自主跑 benchmark |
| 品牌文案調性審查 | ❌ 否 | 需人類品味判斷，不可委外 |
| 合作對象信任評估 | ❌ 否 | 責任無法轉移給 AI 判斷 |
| 策略方向是否正確 | ❌ 否 | 人類護城河所在 |

### 10.4 行動方向 A：讓 AI 複利 — 主動重構可驗證的工作流

1. **分解工作，識別客觀標準** — 系統性地拆解日常工作，找出哪些部分「有客觀成功標準」
2. **設計自主迴圈，而非一問一答** — 讓 Agent 能在有明確回饋訊號的環境中自主運行
3. **你不應該是瓶頸** — 把可驗證的工作授權出去，把時間留給不可驗證的判斷

### 10.5 行動方向 B：守住護城河 — 在不可驗證之處積累判斷力

- **定義問題框架本身** — 不是填充答案，而是提出正確的問題
- **品味、信任與策略方向感** — 這些判斷背後，是你對世界的理解與閱歷，AI 無法複製
- **辨識衡量指標本身的缺陷** — 能夠察覺「我們在優化錯誤的指標」，這是元層次的判斷力

### 10.6 一句話精華

> 那條讓 AI 停止進步的線，同時也是你應該開始積累的那條線。

## 12. Harness Engineering

> summarized from OpenAI Blog — *Harness engineering: leveraging Codex in an agent-first world*

### 12.1 Origin

- 2025 年 8 月，第一個提交由 AI 完成
- 程式碼庫的初始架構、CI 設定、AGENTS.md，全部由 Codex CLI 透過 GPT-5 生成
- **從一開始，就沒有任何人類撰寫的程式碼**
- 五個月後，程式碼庫成長至約**一百萬行**，**1,500 個 PR** 由三位工程師監督完成
- 相當於每人每天 3.5 個 PR

### 12.2 重新定義工程師角色：從編碼到「驅動」AI

| 面向 | 說明 |
|------|------|
| **環境規格化** | 工程師的核心工作變為創建清晰、可強制執行的環境規範 |
| **以提示詞互動** | 人類透過高層次提示詞與系統互動，定義任務，讓 AI 代理執行並自動開啟 PR |
| **AI 自主審查** | 代理不僅自主編碼，也能在本地和雲端環境中互相審查 |

### 提升應用程式的「AI 可讀性」

- **環境分離與 UI 交互** — 應用程式可依據 Git Worktree 啟動獨立執行個體，整合 Chrome DevTools 協定
- **可觀測性工具整合** — Codex 在完全隔離環境中運作，代理可透過 LogQL 查詢日誌、PromQL 查詢指標

### 12.3 將程式碼庫知識設為系統的記錄依據

> 給 Codex 一張地圖，而不是一本 1,000 頁的說明手冊。

#### 為什麼「單一龐大 AGENTS.md」方法會失敗？

1. **稀缺的背景資訊** — 巨大的指令檔案擠壓了任務、程式碼和相關文件
2. **過多的指導即非指導** — 當所有內容都變得「重要」時，就沒有什麼是重要的
3. **即時腐爛** — 龐大的手冊會變成陳舊規則的墳墓
4. **難以驗證** — 單一的大雜燴不利於進行機械式檢查

#### 轉變：AGENTS.md 作為「目錄表」

- 一份簡短的 AGENTS.md（約 100 行）作為地圖，指向其他地方更深入的真實來源
- 知識庫位於結構化的 `docs/` 目錄中

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

- **漸進式揭露** — 代理從小而穩定的入口點開始，被指導接下來該往哪裡看
- **自動化驗證** — 專用的 Linter 和 CI 工作驗證知識庫的更新
- **文件維護代理** — 「doc-gardening」智慧體掃描過時文件，開啟修正用的 PR

### 12.4 代理程式的可讀性：最終目標

- 程式碼庫完全由 AI 代理生成，設計首要目標就是最佳化 Codex 的可讀性
- **負責讀懂完整程式的，是 AI，不是人類工程師**
- **「不存在」的知識** — 任何在執行時無法透過上下文存取的內容（Google Doc、對話紀錄、人類腦中的知識），對系統而言都形同不存在
- **本地程式碼庫為唯一真相來源**

### 12.5 強制規範架構與品味：AI 程式碼庫的基石

- **嚴格的結構邊界** — AI 在具備嚴格邊界和可預測結構的環境中最有效
- **強制分層依賴** — Types → Config → Repo → Service → Runtime → UI，嚴格驗證依賴方向
- **機械式驗證與執法** — 透過自訂的程式碼檢查器、結構測試和 Lint 工具強制執行
- **「品味不變量」的反饋循環** — 人類的審查意見、重構的 PR 以及使用者端的錯誤，會作為文件更新或編碼到工具中

### 12.6 重大挑戰與典範轉移

- **高吞吐量與合併哲學** — 修正成本低廉，等待則代價高昂
- **AI Fullstack 生成** — 從產品代碼到 PR 審查皆由 AI 自主生成
- **自主性水平提升** — AI 現能從單一提示詞啟動，自主完成全流程
- **熵與垃圾回收機制** — 將「黃金原則」編碼入系統，由 AI 代理定期執行清理與重構

### 12.7 Harness 的三大支柱

| 支柱 | 說明 |
|------|------|
| ① **Context Engineering 情境工程** | AGENTS.md 保持短小（~100 行），作為目錄而非百科全書。背景 agent 自動掃描過時文件並開 cleanup PR，實現**文件自我維護** |
| ② **Architectural Constraints 架構約束** | 嚴格的分層單向依賴模型。關鍵洞見：**約束縮小解空間，agent 生產力反而提升**，而非受限 |
| ③ **Observability-Driven Feedback 可觀測性回饋** | Agent 可直接存取 telemetry（logs、metrics、spans），**自主重現 bug 並驗證修復**。環境主動告訴 agent 哪裡出錯 |

### 12.8 對軟體工程的結構性影響

> 「模型是商品；Harness 才是護城河。」

- **瓶頸在 Harness，而非模型** — LangChain 案例：同一模型換用更好的 harness，Terminal Bench 2.0 排名從 Top 30 躍升至 Top 5
- **工程師角色演化** — 從「實作者」轉型為「環境設計者」

#### 四個核心實踐（業界收斂結論）

| 實踐 | 說明 | 對應課程概念 |
|------|------|-----------|
| **Layered Context** | 漸進式揭露情境，agent 只看當下任務所需的資訊 | Harness Engineering + SDD |
| **Agent Specialization** | 以範疇受限的 sub-agent 執行子任務 | Verifiability Mindset |
| **Persistent Memory** | 以 filesystem-backed 狀態跨 session 傳遞進度 | — |
| **Structured Execution** | 固定循環：Research → Plan → Execute → Verify | Eval-Driven Development |

### 12.9 擁抱 Harness Engineering — 工程師的必經之路

> 「這不是新技術的一次性學習，而是一段長期的工程修煉」

#### Harness Engineering 對工程師的核心要求

| 能力維度 | 具體意涵 |
|---------|---------|
| **理解問題本質** | 唯有理解問題本質，才能設計出有意義的約束與驗證標準 |
| **領域深度** | 將模糊需求分解為 agent 可執行的原子任務與可驗證的子目標 |
| **情境優先度** | 在有限 context 中決定什麼資訊最關鍵、何時應換手或介入 |

#### 與大型軟體開發之路的相似與差異

- **相似之處**：架構設計、模組邊界、可維護性、技術債管理，本質問題不變
- **Agent 帶來的新複雜度**：檢測困難（概率性行為）、邊界模糊、情境優先度、最佳化非線性

#### 為什麼長期走這條路反而最有利？

- **組織級工程資產** — Harness 隨時間累積，形成難以複製的競爭護城河
- **正向飛輪效應** — 每次 agent 犯錯 → 修復 Harness → 系統能力提升
- **協同演化關係** — Harness 越成熟，AI 協助越精準

## 13. AI Coding 協作：你變快了，但你學會了嗎？

> Shen & Tamkin (2026) — 一項關於 AI 輔助學習成效的實驗研究  
> 論文：*How AI Impacts Skill Formation*

### 13.1 實驗設計

- 52 名專業開發者，學習全新 Python 函式庫（Trio）
- 隨機分為 AI 組與無 AI 組，進行受控比較實驗

### 13.2 實驗結果

| 指標 | AI 組 | No AI 組 |
|------|-------|----------|
| 完成時間 | 22 分鐘 | 23 分鐘 |
| Quiz 分數 | 📉 低 17% | 📈 顯著較高 |
| Debugging | 差距最大 | 表現最佳 |

> **結論：AI 既沒有讓你更快，也讓你學得更少。**

### 13.3 六種 AI 使用模式

| 模式 | 行為描述 | 時間 | Quiz |
|------|---------|------|------|
| 🔴 AI Delegation | 叫 AI 全寫，直接貼上，不加思索 | 19.5 min | 39% |
| 🔴 Progressive Reliance | 前段自己來，後段全交 AI 處理 | 22 min | 35% |
| 🔴 Iterative AI Debugging | 反覆讓 AI 抓 bug，不自己思考根因 | 31 min | 24% |
| 🟢 Conceptual Inquiry | 只問「為什麼」，自己解決 error | 22 min | 65% |
| 🟢 Hybrid Code+Explanation | 要 code 同時要求 AI 解釋原理 | 24 min | 68% |
| 🟢 Generation→Comprehension | AI 生成後再追問理解，內化知識 | 24 min | **86%** |

### 13.4 為什麼 Error 是學習的燃料

| 指標 | No AI 組 | AI 組 |
|------|---------|-------|
| Error 數中位數 | 3 | 1 |

- Error 強迫理解底層概念
- AI 幫你繞過 Error，也繞過了理解
- 三項核心技能受損最深：**Debugging、Code Reading、概念理解**

> *"I feel like I got lazy. There are still a lot of gaps in my understanding."* — AI 組參與者

### 13.5 如何因應：學習者策略 × 課程設計

#### ✅ 建議做這些

- 先自己 debug，卡很久再問 AI
- 要求 AI 解釋每一行的用途與設計原因
- 用 AI 問「為什麼」，不只問「怎麼寫」
- AI 生成後追問：「這樣設計的原因？」

#### ❌ 避免這些

- 一遇到 error 就立刻丟給 AI 處理
- 直接貼上 AI 輸出，不閱讀不驗證
- 任務完成就以為真的學會了
- 讓 AI 替你做所有 debugging

### 連結課程概念

- **Harness Engineering** — 看不懂程式，就無法驗證 Agent 的行為正確性。程式理解力是監督 AI 的前提。
- **Eval-Driven Development** — 對問題域缺乏深度理解，eval 根本無從設計。知識深度決定評估品質。
- **Vibe Coding ≠ 技能習得** — 快速完成任務不等於真正理解。**AI 是放大器：放大你有的能力，也放大你沒有的空洞。**

> 討論問題：你的使用模式最接近六種中的哪一種？在 HW 作業中，你會如何設計自己的 AI 使用策略，以確保技能真正內化？

## 15. Claude Code 的 Harness 全景

### 六大組件

| 組件 | 檔案 | 功能 |
|------|------|------|
| ① Context Management | `claude-progress.txt`, `feature_list.json` | session 間的工作日誌（短期記憶）+ 任務清單（結構化狀態） |
| ② System Prompt | `CLAUDE.md` | 角色定義 + 恆定規則（系統提示層） |
| ③ Lifecycle Hooks | `settings.json` hooks 配置 | 事件驅動控制層 |
| ④ Sub-agent | `.claude/agents/` | sub-agent 角色定義（.md 檔） |
| ⑤ Skills | `.claude/skills/` | 知識庫注入 |
| ⑥ Slash Commands | `.claude/commands/` | 可重複的 SOP |

另有 `init.sh` — 環境初始化腳本（clean state 保證）

### 15.1 組件 1：Context Management

> 核心問題：如何讓下一個 session 的 agent 不需要重新猜測發生了什麼？

#### 雙 Agent 架構

1. **Session 0：Initializer Agent** — 建立 `feature_list.json`、`init.sh`、progress 模板，做第一個 git commit。不實作任何功能
2. **Session N：Coding Agent** — 執行 `init.sh` 確認環境乾淨，讀取 progress 檔，選一個 `passes=false` 的 feature，實作、測試、commit，更新狀態檔

#### claude-progress.txt 格式範例

```markdown
## 2026-03-24 Session 3
**Status**: In progress
**Last completed**: JWT auth endpoint
**Current task**: Unit tests for auth middleware
**Next session should**: Run npm test, then move to /api/auth/refresh
**Known issues**: Token expiry edge case not handled (line 47)
**Git commit**: a3f9c1d
```

### 15.2 組件 2：CLAUDE.md — Harness 的系統提示

#### 三個核心功能

| 功能 | 說明 |
|------|------|
| **角色定義** | 這個 session 扮演什麼角色、工作範圍在哪裡 |
| **恆定規則** | 專案慣例、禁止事項、溝通協議 |
| **移交協議** | 每個 session 結束前必須執行的動作清單 |

> **防止 early victory 規則**：不得在所有 `feature_list.json` 的 passes 都為 true 前宣告完成。

#### 範例片段

```markdown
# 角色
你是 AIASE-2026 專案的 Backend Engineer
工作範圍：src/api/, src/services/

# 工作限制
- 禁止修改 src/frontend/
- DB migration 必須先 backup

# Session 移交協議
每個 session 結束時：
1. git commit 當前進度
2. 更新 Next session should 欄位
3. 更新 feature_list.json passes
```

### 15.3 組件 3：Lifecycle Hooks

Hooks 是 Harness 的事件驅動控制層：

| Hook | 時機 | 用途 |
|------|------|------|
| **PreToolUse** | 動作執行前 | 守衛：驗證指令安全性、Human-in-the-Loop 確認 |
| **PostToolUse** | 動作執行後 | 自動 lint、執行測試、發送通知 |
| **Stop** | Session 結束時 | 自動更新 progress 檔案、觸發 git commit |
| **Notification** | 需要人工介入時 | 通知開發者、暫停執行等待確認 |

#### guard.py — Human-in-the-Loop 閘道

```python
import json, sys
data = json.load(sys.stdin)
cmd = data.get("tool_input",{}).get("command","")
DANGER = ["DROP TABLE", "rm -rf /", "git push --force"]
if any(p in cmd for p in DANGER):
    print(json.dumps({"decision": "block", "reason": f"危險: {cmd}"}))
else:
    print(json.dumps({"decision": "approve"}))
```

#### settings.json Hook 配置

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{"type": "command", "command": "python3 .claude/hooks/guard.py"}]
    }],
    "PostToolUse": [{
      "matcher": "Write",
      "hooks": [{"type": "command", "command": "bash .claude/hooks/auto_lint.sh"}]
    }]
  }
}
```

### 15.4 組件 4：Sub-agent Coordination

將探索、研究、分析等工作委派給專屬 sub-agent，讓 main context 保持乾淨。

#### Sub-agent 定義範例

```yaml
---
name: code-reviewer
description: 負責 code review，找 bug、安全漏洞、style 問題
tools: Read, Grep, Glob
model: sonnet
skills:
  - coding-conventions
---
你是 code reviewer。
回覆以「問題 / 嚴重度 / 說明 / 建議」格式列出。
禁止修改任何檔案。
```

#### 三種呼叫方式

1. **明確指定** — `@code-reviewer review src/api/auth.ts`
2. **自動觸發** — Claude 判斷適合時自動派送
3. **角色啟動** — `$ claude --agent code-reviewer`

> Sub-agent 的工具集應嚴格限制在其職責範圍內。

### 15.5 組件 5：Skills（知識庫注入）

解決「每次都要重新解釋專案慣例」的問題 — 把領域知識寫成 skill 檔，在 agent 啟動時直接注入 context。

#### Skills 目錄結構

```
.claude/skills/
├── api-conventions.md   ← REST 格式與編碼規範
├── db-patterns.md       ← 資料庫操作規則
└── testing-strategy.md  ← 測試策略
```

#### Skills vs CLAUDE.md 的差異

| 面向 | CLAUDE.md | Skills |
|------|-----------|--------|
| 作用範圍 | 所有 session 恆常載入 | 特定 agent 啟動時注入 |
| 內容性質 | 角色定義、規則 | 知識、模式、checklist |
| 適合放 | 「你是誰、你的規則」 | 「你需要知道的領域知識」 |

> Skills 讓知識模組化、可重用。同一份 `api-conventions.md` 可以被多個 agent 共享引用。

### 15.6 組件 6：Slash Commands（可重複的 SOP）

把常用的多步驟流程打包成一個指令。

#### .claude/commands/continue.md

```markdown
---
description: 繼續上次的工作
---
執行以下 SOP：
1. bash ./init.sh
2. 讀取 claude-progress.txt，了解上次狀態
3. 讀取 feature_list.json，找一個 passes=false 的 feature
4. 實作 feature（做一個就好）
5. 以 end-to-end 方式測試（使用者操作視角）
6. 更新 feature_list.json passes=true
7. git commit，更新 progress 檔
```

使用方式：`$ claude` → `> /continue`

> 一個指令，七個步驟，每次執行結果一致。Commands 是 Harness 可重複性的核心保證機制。

## 16. Harness Engineering 的三層金字塔

三者的遞進關係（不可跳級）：

| 層級 | 名稱 | 說明 |
|------|------|------|
| 底層 | **Prompt Engineering** | 設計單一 prompt 的措辭與結構，是所有上層能力的基礎 |
| 中層 | **Context Engineering** | 設計如何在正確時機、以正確格式把資訊傳給模型 |
| 頂層 | **Harness Engineering** | 設計 agent 的整合點（commands、hooks、skills、agents、MCP），最靠近「系統工程」 |

> 不懂 Prompt Engineering，就無法做好 Context Engineering；  
> 不懂 Context Engineering，就無法做好 Harness Engineering。

---

## 17. 設計原則：少即是多

| 原則 | 說明 |
|------|------|
| ⚒️ **最小工具集** | 每個角色只拿它需要的工具。reviewer: Read, Grep, Glob（非 Write 或 Bash） |
| 📋 **明確移交協議** | 下一個 agent 不用猜，所有狀態都在結構化檔案中，30 秒即可上手 |
| 🔂 **一次一件事** | Incremental progress 優先於 one-shot。每個 session 只完成一個 feature |
| 🧹 **Clean State 保證** | 每次結束要能讓任何人接手。`init.sh` 確保環境重現性 |
| 🗂 **結構化狀態** | 用 JSON 而非 Markdown 記錄機器要讀的資料。模型對 JSON 更「尊重」，不易擅自刪改 |

## 18. No Vibes Allowed：在複雜 Codebase 中解決困難問題

> 整理自 Dex Horthy（HumanLayer）在 AI Engineer 峰會的演講  
> Dex 是 12-Factor Agents 框架提出者，也是 Context Engineering 概念的早期推廣者

### 18.1 為什麼 AI 寫程式會產生 Slop？

- **問題規模** — 10 萬名開發者調查：AI 確實提升程式產出量，但工程師花大量時間修復 AI 產出的低品質程式碼（slop）
- **Greenfield vs. Brownfield** — 全新專案中 AI 表現亮眼；面對複雜舊有 codebase，AI 往往製造大量 slop 與技術債
- **Dex 團隊的反例** — 三人團隊、八週時間，達到 2-3 倍產出且品質不打折。差異不在工具，而在方法論

### 18.2 LLM 的本質：Stateless + Context 決定一切

> Better tokens in, better tokens out.

| 面向 | 說明 |
|------|------|
| ✅ **正確性** | Context 中絕對不能有錯誤資訊。一個謊言會污染整個推理鏈 |
| 📋 **完整性** | 不能缺少關鍵資訊。模型無法推斷它從未看過的邏輯 |
| 📏 **大小** | 不能塞太多噪音。過量的無關資訊會稀釋重要信號 |
| 🧭 **軌跡** | 對話的方向感至關重要。一直糾錯會讓 AI 學習到「繼續犯錯」的模式 |

### 18.3 笨蛋區（The Dumb Zone）

- Context window 使用量超過約 **40%**，模型表現就會明顯下降
- 模型並非突然罷工，而是悄悄地開始遺漏細節、忽略約束、輸出看似合理卻錯誤的程式碼
- ⚠️ 常見陷阱：裝太多 MCP server，光是工具定義本身就能把 context 直接推進笨蛋區

```
聰明區（高品質輸出）    分界點（臨界點）    笨蛋區（品質下降、slop 增加）
0% ─────────────────── 40% ──────────────────── 100%
```

### 18.4 核心工作流：Research → Plan → Implement (RPI)

> 核心目標：讓 context window 持續待在聰明區

#### 🔍 Research 階段

- 用 sub-agent 對 codebase 做**垂直切片式探索**，而非全面掃描
- 目標是產出一份基於實際程式碼的研究文件
- 靜態文件會過期；唯有直接讀取原始碼，才能得到當下真實的系統樣貌
- 第一輪 research 品質不佳？**直接丟掉重做**，不要試圖在爛的基礎上修補
- 💡 Research 文件是整個流程最重要的輸入，也是最值得人工驗證的環節

#### 📋 Plan 階段

- Plan 的品質標準：**具體到「全世界最笨的模型也不太可能搞砸」**
- 包含精確的檔名、行號、程式碼片段
- 不是「修改 auth 模組」，而是「修改 `src/auth/token.ts` 第 47 行的 expiry 邏輯」
- Plan 是人與 AI 之間的**契約**。越精確的契約，執行時越少需要臨時決策
- 技術主管的審查點：讓技術主管讀 plan 就能掌握系統方向。這是人類精力**最高槓桿的介入點**

#### ⚙️ Implement 階段

- 照計畫執行，保持 context 最小化
- 實作階段不是思考階段 — 思考已經在 research 和 plan 中完成了
- 只餵入 plan 明確指向的檔案與片段
- 遇到意外狀況，先停下來更新 plan，再繼續實作
- 不讓 agent 在實作中途自行做 research

### 18.5 刻意壓縮（Intentional Compaction）

- 隨著對話延伸，context 不可避免地累積噪音
- 解法：**隨時讓 agent 把當前 context 壓縮成 markdown 文件**，人工 review 後開新 context 繼續
- 壓縮文件應包含：當前任務狀態、尚未完成的步驟、相關檔案與行號、具體問題描述與已知限制

#### 謊言密度四層

```
原始碼（最真實）→ 函式名稱 → 註解 → 文件（最容易過期）
```

> 壓縮的是「真相」，不是靜態文件。  
> 壓縮過程本身就是一次系統理解的驗證機會。

### 18.6 Sub-agent 的正確定位

> Sub-agent 是控制 context 的工具，不是角色扮演的演員。

#### ❌ 錯誤用法

- 擬人化角色分工（frontend agent、backend agent、QA agent）
- 讓主 agent 做所有搜尋、理解與探索，導致 context 爆炸
- 讓多個 agent 共享同一個龐大 context

#### ✅ 正確用法

- Fork 出全新的獨立 context，專門執行探索任務
- Sub-agent 只回傳精簡結論，不把探索過程帶回主 context
- 將「產生噪音的工作」隔離在子 context 中，主流程保持乾淨

### 18.7 不要外包思考

> AI 不能取代思考，它只能放大你已經做過（或沒做過）的思考。

| 層級 | 錯誤類型 | 影響 |
|------|---------|------|
| 1 | 錯誤的 code | 一行問題，容易發現、容易修正 |
| 2 | 錯誤的 Plan | 一百行問題，需要大量返工，代價高昂 |
| 3 | 錯誤的 Research | 對系統的根本誤解，導致整個下游全部歪掉，**最致命** |

> 人類精力應集中在**最高槓桿的環節**：確認 research 和 plan 是否正確，而不是逐行審查 AI 生成的程式碼。

### 18.8 任務複雜度光譜

| 任務規模 | 建議做法 |
|---------|---------|
| 改個按鈕顏色 | 直接跟 agent 說，無需任何前置工作 |
| 小功能開發 | 撰寫簡單 plan，確認邊界與輸出格式即可 |
| 跨多 repo 中型功能 | 完整 Research + Plan，再進入 Implement |
| 最複雜任務 | 完整 RPI，多輪迭代，每輪結束後做 Compaction |

> 💡 怎麼知道該用多少 context engineering？靠練習。先選一個工具，認真練到精通。

### 18.9 團隊文化才是真正的難題

| 角色 | 現象 |
|------|------|
| **資深工程師** | 傾向不採用 AI，覺得沒快多少。結果：每週花大量時間清理他人產出的 slop，對 AI 越來越排斥，形成惡性循環 |
| **中階工程師** | 大量使用 AI 工具，填補技能落差，產出量顯著提升。但缺乏系統性方法論，容易產生大量 slop |

> **根本問題**：這不是 AI 的錯，也不是個別工程師的錯 — 這是**文化變革問題**，需要從上層推動，建立共同的方法論與品質標準。
